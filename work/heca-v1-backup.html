01 — Overview

No design system. A CRM 40 people couldn't stand. A blank canvas.

Heca's internal CRM had grown organically for years. Every screen had been built ad-hoc — 200+ inconsistent patterns, no shared vocabulary between design and engineering, and three distinct user roles (admin, manager, operator) all forced through the same generic interface.

I owned the work end-to-end: from human-centered research, to a token-first design system, to documented production handoff with the remote engineering team.

Constraints

No existing design system. Fully remote team. Three user roles with conflicting needs and permissions.

Approach

Token-first. Build the system primitives before any screen, so every decision compounds instead of fragmenting.

02 — The Problem

Inconsistency wasn't cosmetic — it was operational.

The same action looked different on every screen. Operators second-guessed which button saved and which discarded. Engineers rebuilt the same table four times because nothing was shared. Each new feature widened the gap.

03 — Research

Interviews with 40+ users, then an audit of everything that existed.

I ran user interviews and task analysis with the full base of 40+ internal users across all three roles, then audited every existing screen — cataloguing the 200+ patterns into a single inventory so the scale of the inconsistency was undeniable to stakeholders.

A

Roles need different defaults, not different apps

Admins, managers and operators shared 70% of tasks — the friction was in surfacing the right 30%, not in building separate products.

B

Trust eroded at the save/discard moment

Ambiguous, inconsistent actions made users hesitate on every record — the single biggest source of self-reported friction.

C

Engineering wanted constraints, not pixels

The remote team needed documented tokens and rules they could build against — not one-off redlines per screen.

04 — UX Thinking

Token-first: build the system before the screens.

Rather than redesigning screens and hoping consistency would follow, I started with the primitives: semantic color, a spacing scale, type ramp, and component variants with explicit states. Every screen was then composed from this vocabulary — so improvements compounded instead of fragmenting.

01 Tokens

Semantic colors, spacing scale, radii, and type ramp defined as named tokens — one source of truth.

02 Variants & States

Every component built with auto-layout, full state coverage, and WCAG 2.1 contrast baked in.

03 Role-aware views

One interface, smart defaults — the right tools surfaced per role instead of three separate apps.

04 Documented handoff

Token-to-code parity, documented in-Figma so engineering could build with confidence.

05 — UI System

A Figma library that maps one-to-one to production.

The system shipped as a documented component library — variants, states, auto-layout, and WCAG 2.1 compliance — with token-to-code parity so design and engineering finally spoke the same language.
